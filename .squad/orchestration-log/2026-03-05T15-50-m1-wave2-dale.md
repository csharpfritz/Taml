**ISO 8601 Dates:** ⚠️ **MINOR DISCREPANCY**
- **D1: YYYY-MM format (year-month)**
  - Spec requires: `YYYY-MM` is minimum date pattern (e.g., `2024-01`)
  - .NET: Correctly detects `2024-01` as DateTime ✅
  - Python: Treats `2024-01` as string ❌
  - JavaScript: Treats `2024-01` as string ❌
  - **Impact:** Medium — Python and JS miss spec-required format
  - **Recommendation:** Add YYYY-MM detection regex to Python and JavaScript

**Duplicate Bare Keys:** ❌ **CRITICAL DISCREPANCY**
- **D2: Nested duplicate bare key structure**

For TAML input:
```
games
	game
		home	Philadelphia
		away	Dallas
	game
		home	New York
		away	Boston
```

| Parser | Result | Spec Compliant? |
|--------|--------|----------------|
| JavaScript | `{games: [{home: Philadelphia, away: Dallas}, {home: New York, away: Boston}]}` | ✅ |
| Python | `{games: {game: [{...}, {...}]}}` | ❌ |
| .NET | `{games: {game: [{...}, {...}]}}` (probable) | ❌ |

**Issue:** Spec Rule 4 states "When a parser encounters duplicate bare keys at the same level, it should convert the **parent** to a Collection of Objects." JavaScript correctly converts `games` to a list. Python and .NET keep `games` as dict and put collection under `game`.

**Impact:** HIGH — Applications switching parsers get incompatible data structures for identical input.

**Recommendation:** Team decision required:
1. **Option A (Spec Literal):** Modify Python and .NET to match JavaScript behavior (drop intermediate key)
2. **Option B (Practical):** Update spec to allow intermediate key (better for serialization round-trips)
